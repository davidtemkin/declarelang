# Documentation governance

**Status:** RATIFIED-IN-PROGRESS, 2026-07-14 (David + Claude). This is the contract the
documentation is built and maintained against. It **supersedes** the two stalled drafts
[`doc-system.md`](doc-system.md) (the generated-reference/`@api`/gate design) and
[`documentation-plans.md`](documentation-plans.md) (the doc-block authoring side) — both
fold into this one governance doc; their still-live mechanics are absorbed below.

This is itself a **category-A** document (see §1) — internal, about how docs work, not a
doc users read. It lives in `docs/system-design/` — the single home for the whole system's
design record.

---

## 0. Why this exists

The docs were three parallel copies of the truth with no binding to the compiler and no
graduation path:

- the language **spec** (`declare-language.md`),
- the **LLM brief** (`declare-for-llms.md`) — hand-maintained, generated by nothing,
- the **guide** (`docs/guide/`) — hand-written narrative,

each free to drift from the others and from the compiler. The verify/evals work proved
the gap concretely: the first eval cycle's top finding (E-1) was a *docs gap* — the brief
never showed `stroke()` — patched by hand, reactively, with no mechanism to keep it from
recurring. Docs were unverified truth. This document fixes that structurally.

## 1. Two categories — the governance backbone

Every documentation artifact is exactly one of:

- **A — System design (the record).** Internal, contributor-facing, **non-authoritative**:
  background and reasoning, not truth. The design discussions and record for **every facet
  of the system** — language, components, runtime, tooling — plus the historical decision
  record, the eval E-series, roadmaps, and this doc. May be speculative, dated, incomplete,
  historical. **Lives under `docs/system-design/`** — one home for the whole system's design
  record; never at the distro top level. (Truth is category B and the compiler; A is *why*,
  not *what is*.)
- **B — Knowledge.** For *users*, human **or** LLM. **Authoritative.** The prose is
  authored once; a human and an LLM read the *same words*; only the packaging and access
  differ (on-disk file nav, the docs app, injected LLM context, a future editor/IDE).

**The line runs through rationale, not around it.** Rationale that helps you *use* the
language is B and lives in the core doc, terse ("the comma is a terminator, Go's rule, so
the last line needs no special case"). Rationale about the *implementation* or *unsettled
design*, and the deep historical record, is A. Keeping that line is what makes the one
authoritative core doc both complete and small enough to hold in your head.

## 2. The category-B artifacts

| artifact | function | authored / generated | held true by |
|---|---|---|---|
| **Core doc** — spec + usable rationale + **the LLM brief** | read-through; holds the *whole language model*; the "fits in your head / in the context window" artifact; the doc an eval hands the model | hand-authored, small **by design** | executable examples + surface cross-checks (§4) |
| **Reference** — exhaustive per-element lookup (every attribute, method, component, diagnostic) | random-access, not read-through | **generated** from `@api` doc-comments in the source | the gate: coverage/orphan/structural (§4) |
| **Guide** — foundational concepts + tutorial | first-class **narrative path** through Declare | hand-authored, **derivative** (§3) | backing links + the gate (§4) |
| **Operational** — getting started, dev server, production build, the CLIs (`verify`/`format`/`declarec`), flags, the playground | how to *run* the toolchain | registry-derived where a registry exists; else hand-authored + **executable in CI** | the source registry, or a CI smoke-test |

The core doc is *not* the guide: the core doc is the complete surface stated once; the
guide is the expansive path that teaches it. Both are first-class; both stay true.

### 2.1 One artifact, and it must work for LLMs (RULED 2026-07-14, David)

There is **one** language artifact, the core doc, and it carries the LLM-brief role — not a
separate `declare-for-llms.md`, not a derived twin. The binding constraint is
**LLM-effectiveness**: the single doc must serve a model generating from it *at least as
well as the best purpose-built brief we have*, measured by the evals (§4).

This is a live acceptance criterion, not an assumption. A head-to-head (Sonnet one-shot,
n=3, `evals/harness --brief-doc`) measured the current unified `declare.md` **worse** as a
generation context than the incumbent lean `declare-for-llms.md` — **0/9 vs 2/9 green, 531K
vs 463K tok**. Two located core-doc bugs were fixed in the pass (the `name(): T {}` method
footgun → §3 negative rule; intrinsic-shadowing → the sizing section) and *did* land — the
footgun vanished from all drafts and failures climbed from R1 syntax to R4/R5 — but the
gap is **packaging density**, not those footguns: the lean brief front-loads rules, states
negative knowledge inline, and keeps examples tight, and that wins per-token.

**Consequence for the alignment pass (§8):** the core doc must **absorb those packaging
techniques** — not just contain the spec, but present it the way the winning brief does.
`declare-for-llms.md` is **folded, not deleted-blind**: it stays as the eval's parity
baseline and the harness default (`--brief-doc`) until the core doc **measures ≥ it**, and
only then is it retired. LLM-effectiveness parity is a **DoD gate on the core doc** (§6),
enforced by the teaching gate (§4).

**Where verify/evals docs sit:** *how to verify a program* (the ladder as a concept) →
**Guide**; the `verify`/eval **CLIs** → **Operational**; the **diagnostics catalog**
(`DECLARE####`) → **Reference** (it is a registry). The eval **findings** — the E-series, the
harness design, RESULTS interpretation — are **category A**.

## 3. The derivation chain — truth flows up from the code

```
sources (@api doc-comments) ──▶ Reference ──▶ Guide
Core doc: hand-authored, gated by executable examples + surface cross-checks
Operational: registry-derived (flags, diagnostics) or executable-in-CI
```

The **guide is derivative but original**: every *fact* it states links up to the
reference or core doc that backs it (§4); what it *originates* is the path, the conceptual
framing, and the sequencing. "Derivative" ≠ "generated" and ≠ "adds nothing."

### Two ways docs live beside code

1. **System components** — the standard library, the runtime, the language surface. These
   are *part of the system*, documented **as** system: `@api` doc-comments inline in the
   source → generated reference + guide coverage. **The standard library is here** — it is
   system surface (auto-included, themed, first-class), not example code. Its user-facing
   content is `@api` prose → reference + a guide chapter; its design *contract/rationale*
   (e.g. `components-baseline`) is category A → `docs/system-design/` (the system's design
   record, not scattered beside the code).
2. **Bundled apps** — `apps/<app>`. Demonstration artifacts, *consumers* of the
   language, not part of it. Their specs/briefs live in `apps/<app>/` as that
   artifact's own documentation and **never flow into the reference**.

## 4. Truth-maintenance — the gate (all CI-blocking)

- **coverage** — every `@api` element has prose (a doc-comment or a prose file).
- **orphan** — every prose entry maps to a real `@api` element.
- **structural** — the generated model matches the extracted surface.
- **dangling links** — every `declare:` link (§5) resolves to a real target.
- **backlink coverage** — reference nodes with no inbound guide link are flagged holes.
  *(This is what would have caught the un-taught component library.)*
- **executable examples** — every runnable fence in a category-B doc compiles **and**
  behavior-asserts where it claims behavior (reuse `verify` rung 5).
- **surface cross-checks** — attribute / component / diagnostic / flag names in prose are
  validated against the schema, registry, `Diag` catalog, and `FLAG_SPECS`.
- **evals as the teaching gate** — "can a model do the task from the docs alone?" A
  docs-gap E-series finding is a prose bug with a location.

## 5. Linking — the connective tissue (author once, resolve many)

**Scheme:** `declare-docs:` — deliberately its own namespace, walled off from a future
general Declare app-router (symbolic target → location + browser back/forward), which will
claim its own scheme and must not be constrained by the docs system. Resolving a
`declare-docs:` link is itself a small routing problem; the docs-app resolver is where that
muscle first gets built, but the app-router design stays open.

- **Symbolic IDs, never file paths or heading text.** `View.x`, `Slider.value`,
  `DECLARE2000`, `concept:reactivity`, `spec:scope-nouns`, `guide:composition`. Files and
  headings move freely *under* the ID; the ID is the contract.
- **IDs are generated, never hand-created** — this is a hard rule: *no human ever assigns
  an ID*, so there is no ID-creation mistake to make.
  - *Reference / diagnostics / flags:* the ID **is** the source symbol — `View.x` =
    `Class.member`, `DECLARE2000` = the `Diag` code, a flag ID = its `FLAG_SPECS` key.
    Extracted, mistake-proof by construction. (Already the `model.json` node keys.)
  - *Prose sections* (core doc, guide, operational): the ID is auto-slugged from the
    heading and then **pinned** — editing the heading text does not silently reassign the
    ID, which is what makes it stable *and* automatic. The author never types an ID.
- **One authored scheme, many resolvers:** `[the setter](declare-docs:concept:reactivity)`.
  Each packaging resolves it its own way — on-disk markdown → relative path + anchor; docs
  app → node navigation (with history); LLM context → a retrievable handle / inline
  expansion; a future editor → go-to-definition / hover. Same prose, same links, different
  access.
- **The only hand-authored part is the link target choice**, and the **dangling-link gate**
  (§4) guarantees a wrong target *cannot ship* — a typo fails the build, it never rots
  silently. Editor tooling later turns link-authoring into autocomplete, closing even the
  typo window; the gate makes it safe on day one.
- **A guide fact with no backing link is a lint** — the link *is* the evidence the
  derivative layer is true.

## 6. Definition of Done

A change to the language surface is **not done** until the category-B docs reflect it and
the gate passes — the same standing as *format-to-canon* and *verify*. Docs are a DoD gate.

## 7. Distro layout

- **`docs/` = the one documentation root** (category B): the core doc, `guide/`,
  `reference/`, `operational/`.
- **`docs/system-design/` = category A** (this doc + the merged former top-level `design/`
  and `docs/system-design/`). One internal-docs directory for the whole system's design record —
  named `system-design/` (not `language-design/`) because it spans every facet: language,
  components, runtime, tooling. The top level loses `design/` and `docs/system-design/` and gains
  nothing; `README.md` is the entry point and links onward.
- **Artifact specs co-locate with their artifact:** `site-spec.md`/`site-brief.md` →
  `apps/site/`, `weather.declare` → its example, the tabslider notes → that component.
- **The standard library gets system docs** (§3 mode 1), not bundled-app treatment.

## 8. Sequence

Two regimes, opposite orders — do not apply one regime's rule to the other.

### Bootstrap — out of deep drift (RULED 2026-07-14, David): **align first, gate second**

The corpus is drifted deeply enough that gating first would fence a construction site — the
gate's targets (the merged core doc, the `declare-docs:` link graph, the coverage set, the
`@api` prose) don't exist until the rewrite creates them, and the real shape of the checks
is learned *by doing* the alignment. So:

1. **Ratify this contract** — it is the *target* the rewrite aims at (not a fence).
2. **(After the distro reorg settles.) The alignment pass** — a thorough
   *update-and-rewrite-as-needed* of the whole corpus *into* this structure: merge spec +
   brief into the core doc, write `@api` prose for the reference, write/repair the guide,
   adopt `declare-docs:` links + stable IDs as you go. Rewrite to be **gate-clean** so the
   gates later *lock* rather than re-open the work. The compiler + `verify` + evals are the
   **manual truth-oracle** throughout — run by hand, before they're automated. Keep it
   finite with an up-front audit (per topic: keep / update / rewrite / write-new + missing
   list) and drive it **core-doc-first → reference → guide**.
3. **Turn the gates on** (the ramp, §9.2) to *hold* the aligned corpus. From here they
   prevent regression.

The read-only audit may start now; the rewrite waits for the reorg (both touch `docs/`).

### Steady state — after alignment: **mechanism holds**

The DoD rule (§6) + the gates. A surface change updates the docs in the same commit and the
gate holds. *Here* the "don't rewrite ahead of the mechanism" caution applies — but only
here, not to the one-time bootstrap out of drift.

   the future app-router). IDs auto-generated, never hand-created (§5).
2. **Gate ramp** — *when* each §4 check becomes CI-blocking (not whether it exists). Rule:
   run everything in report mode from day one; flip a check to blocking only once the docs
   can pass it; each check is permanent once on. Proposed stages:
   - **0 advisory** — all checks report, none block (measures the backlog).
   - **1 structural (block ~now)** — structural, orphan, dangling-links, fences-compile.
     Cheap, deterministic, mostly-already-true. *This is the §8 "mechanism".*

     verify rung 5) + surface cross-checks (names vs schema/registry/`Diag`/`FLAG_SPECS`).
   - **3 completeness (block after the rewrite)** — coverage + backlink coverage. Can't be
     blocking earlier — they gate *on* the rewrite being done.
   - **4 teaching (periodic, not a per-commit wall)** — evals-as-gate: non-deterministic +
     budgeted; a tracked quality bar, not red/green CI.
   Independent of the distro reorg (checks are about content + links, not layout) → ratify
   now, build after the reorg settles.
3. ~~`components-baseline.md` rationale home~~ — **RESOLVED: `docs/system-design/`** (the
   system design record, all facets). User-facing component content: `@api` → reference +
