Just as SQL is a domain-specific language for database queries, Declare is a DSL for user interfaces. Its whole surface fits in your head — and inside an LLM's context window.

Why a new language, now? →

01

Read it. Generate it. Run it.

The relationships that matter — structure, data, reactivity — are explicit in the language, not hidden in imperative steps or a runtime graph.

Analyzable

A program's whole structure — components, data, what reacts to what — is right there in the source. The compiler reads it statically; so can an LLM.

Predictable

One clear way to write each thing, and the compiler rejects the rest — no dependency arrays, no keys, no stale-value bugs. An LLM's mistake is a compile error that names the fix — not an app that quietly runs wrong.

Runnable

The compiler runs in the browser, so code renders the moment it exists — no build, no deploy. Write it and it's live.

Everything that makes it legible to an LLM makes it legible to you.

02

Small. Fast. A whole app.

Declare Calendar: four views that are one surface seen through a moving, zooming rectangle — event chips that morph into time-blocks, drag-to-reschedule, an editable detail panel, light and dark. The kind of continuous, animated UI that's usually a project of its own — here it's the reference app.

Holds framerate

Reactive dependencies are compiled, not tracked at runtime, and motion is declarative springs — so the continuous zoom stays smooth, with no animation library and no reactive-graph tax.

Type-checked bindings

Events, selection, the editable panel: the data flow is declarative, type-checked bindings, not a state manager wired by hand. Change a value and everything derived from it follows — a type mismatch is a compile error, not a runtime surprise.

Easy to reason about

A zooming, drag-to-reschedule, edit-in-place calendar is normally a thicket of imperative view-syncing and animation state. Declared, it's a fraction of the code — and legible to you, the compiler, and an LLM alike.

03

Declare it. It runs. It stays true.

A binding isn’t a callback you remember to re-run — it’s a standing relationship the runtime keeps true, re-derived the moment its inputs change.

Three bindings track one value: the big number, the bar’s width, and its color. Click to bump v — all three update together, and the color crosses cool to warm at the halfway mark.

04

Declare the shape, not the loop.

A view can be built from data instead of laid out by hand — the shape written once, the data deciding how many there are and what each one says.

One Bar, instanced once per row of the data. Edit a number or add a row and the list rebuilds itself — no loop, no keys. And Bar is auto-included: no import, no definition, just use it.

05

Motion you declare, not schedule.

You say where a thing belongs; a Spring finds the path there and settles by physics — motion you declare, not a timeline you sequence.

Click to toggle the ball’s target between two positions. The Spring follows and settles by physics — no timeline, no frames driven by hand.

06

Declared states, not manual wiring.

A State is a named bundle of overrides — a whole configuration you can apply as one. The mode is the unit of change, not each property by hand.

Click to flip one boolean. The card’s height, its color, and an extra line of text swap in together — and revert the instant the mode turns off.

The web’s last generation of frameworks was built on the browser’s document model. Declare abstracts the renderer away — you declare what the UI is, and the runtime draws it.

The same program paints to the DOM or to a canvas today, WebGL next, with no rewrite. And you never touch the render layer — no DOM to poke, no paint loop to run — so the code stays small: what you write is the interface itself, not the machinery that shows it.
